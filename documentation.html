Here you can find a list of core libraries, specifications and coding standards that we use in aap.

AngularJS

AngularJS is the core of aap. It is not a traditional html template, it's an AngularJS app. We strongly recommend checking the AngularJS before start doing anything with aap.

Angular Material

Angular Material is a set of AngularJS directives and services that implements Material Design Specifications and it's in active development by Google.

John Papa's Angular Style Guide

We are following John Papa's Angular Style Guide to create common patterns throughout the entire application.

Yeoman generator for AngularJS with GulpJS

We are using generator-gulp-angular package for quickly bootstrapping the application. You can find more information about the generator in the Prerequisites page.
Note: this section will give you some information about what tools you will need. You can skip to the Installation section to start installing the template. We already mentioned all the prerequisites and how to install them in the Installation section.

Yeoman generator for AngularJS with GulpJS

aap uses Yeoman generator for AngularJS with GulpJS for quickly bootstrapping the application. You can check out the link for more detailed information. Simply, Yeoman generator for AngularJS with GulpJS will automatically inject any templates, controllers, modules, directives and other application related files into the index.html file for you. It will also automatically find and compile scss files into one css file and inject that one as well.

Node.js

To install and use aap, you will need Node.js installed to your computer. We won't get into too much detail about Node.js as it's out of the scope of this documentation. Also you won't need to actually use Node.js, it's only required for the development process.

Git

To be able to install and use aap, you will also need Git installed to your computer. Git is required for bower to work correctly.

Bower

aap uses Bower package manager to install and manage 3rd party components and libraries.

Gulp

Gulp is required for aap to work properly. Although there are ways to develop without using Gulp or Node.js, we strongly recommend you to use them.

Installing Prerequisites

  1. Download and install the latest Node.js from its web site.
  2. Download and install the latest Git from its web site.
  3. Open your favorite console application (Terminal, Command Prompt etc.), run the following command and wait for it to finish: npm install -g bower
  4. Run the following command and wait for it to finish: npm install -g gulp
Now you are ready to install the aap.

Installing aap

  1. Unzip the aap source code zip file
  2. Extract the contents of the zip file into a folder that you will work within. For this documentation, we will refer that as "your work folder".
  3. Open your favorite console application (Terminal, Command Prompt etc.), navigate into your work folder, run the following command and wait for it to finish: npm install This command will install all the required Node.js modules into the node_modules directory inside your work folder. Note: Installing Node.js packages may throw a lot of warnings and errors along the way. As long as the process finishes without any error notes such as "Killed", you will be fine.
  4. Run the following command and wait it for to finish: bower install This command will install all the required bower packages into the bower_components directory inside your work folder.
And now, you are ready to run the aap for the first time.

Running aap

  1. While still in your work folder, run the following command in the console application: gulp serve Gulp will take care everything and start the aap. Your default browser will be opened automatically, and you will be able to navigate through the application.

Demo Structure

Here you can see the directory structure of the aap:

app/core

Contains all the core elements that required for aap to work. It's highly recommended NOT to touch or modify anything inside the core directory so you can easily update the template when a new version released.

app/data

Contains all the json files that holds demo data. We are using json files to emulate API calls and the data flow. These json files are good for seeing the data structure we are using in our applications so you can use similar data structure to minimize your work.

app/main

Contains all the modules. Apps, Components, Custom Pages and everything else you can see in the aap are located in this directory.

app/sidenav

Contains the sidenav modules.

app/toolbar

Contains the main toolbar module.

index.***.js

These files are the entry point of the main application. aap is an Angular module by itself. It means you can add aap as a module into another project.

index.scss

This is the main scss file. DO NOT REMOVE it as all the other scss files will be injected into this file before compiled to a css file.

Adding a Module

To add a new module, simply put your module files into your preferred location inside the app/ directory and then add the module name as a dependency into the app/index.module.js file. You don't need worry about injecting any of your module files into anywhere else as Yeoman generator for AngularJS with GulpJS does that automatically. It will automatically find your controllers, directives, html templates, css files etc. and it will inject them into correct locations.

Removing a Module

Removing a module is very easy too. All you have to do is to locate the module name from dependency list inside the app/index.module.js file and remove it. Then you can remove the actual module files. Just like adding a new module, you don't have to worry about any leftover injectors as they were automatically injected by Yeoman generator for AngularJS with GulpJS.

Third Party Libraries

aap uses Bower package manager to manage third-party libraries. Combined with the generator, you can add, remove or update third-party libraries in a matter of seconds.

Adding a Third-Party Library

To add a third party library to your application, open your favorite console application (Terminal, Command Prompt etc.), navigate to your work folder and run the following command by replacing the PACKAGE_NAME with the actual name of the package you want to uninstall: bower install PACKAGE_NAME --save If you are installing an AngularJS module, then you have to add it as a dependency to your application. You can do that either in /app/core/core.module.js file or in /app/index.module.js file.

Removing a Third-Party Library

To remove a third party library from your application, open your favorite console application (Terminal, Command Prompt etc.), navigate to your work folder and run the following command by replacing the PACKAGE_NAME with the actual name of the package you want to install: bower uninstall PACKAGE_NAME --save If you are uninstalling an AngularJS module, make sure you remove it from your dependencies lists as well. You can check both /app/core/core.module.js file and /app/index.module.js file to make sure you remove the dependency injections. Note: Other than adding or removing package names to or from the dependency lists, you shouldn't need to anything else to include the actual library into your application. Just reload your server and everything will be injected automatically. Note: Sometimes, bower packages might have dependencies to other packages. Normally, uninstalling a package will remove its dependencies only if those dependencies are not in use. If you find yourself with lots of unnecessary packages in your bower_components folder, running bower prune command will remove all the unnecessary packages. This should only uninstall the packages that are not in use, but you should still be careful. Bower will output the names of the removed modules to the console, so pay attention when you run the above command and make sure it didn't remove any important packages.

API

aap comes with 2 different methods for APIs. They are nothing more than convenient ways of storing and calling $resource objects. If you want to keep all your API endpoint definitions separated from each other than this is the suitable method for you. In aap, all the applications and pages that requires a simulated API call are using this method. Mainly because with using this method, we are following the modular structure of the aap. Every module can live by itself and removing one will not affect others or will not left any unnecessary codes behind. This method requires the msApi service which can be found in /app/core/services/ms-api.provider.js file.

Defining API Endpoints

To register your API endpoints, you can use the following method in the config blocks of your modules: /** @ngInject */ function config($stateProvider, msApiProvider) { // State $stateProvider.state('app.somepage', { url : '/somepage', views : { // ... }, resolve: { PageData: function (msApi) { return msApi.resolve('app.somepage@get'); } } }); // Api msApiProvider.register('app.somepage', ['http://myapi.com/pageData']); } If we look at the above example, you can see that we registered our API endpoint using the msApiProvider.register() method and then resolved it in the state definition by using the msApi.resolve() method. Keep in mind, msApi and msApiProvider are essentially the same service. It has both provider and a normal service so you can inject it into Config blocks as well as into Controllers and other Services.

Register

register method is available in both msApi and msApiProvider. msApiProvider.register(Name, Resource); msApi.register(Name, Resource); The resource parameter is essentially the exact same thing that you would put into a $resource service. So lets say you have the following $resource definition: $resource(url, paramDefaults, actions, options); You would simply put definitions inside the resource array just like you were defining a $resource object: msApiProvider.register('getTaskboard', [url, paramDefaults, actions, options]); Basically, we moved everything from inside the $resource definition into the resource array. The paramDefaults, actions and options are optional just like they are in the $resource.

Request

To call the APIs in Controllers and other Services, you can use the request method. It is only available in msApi service. msApi.request(Action, Parameters, Success, Error); /** @ngInject */ function TaskboardController(msApi) { var vm = this; // Data vm.taskboardId = 3; ////////// msApi.request('getTaskboard@get', {id: vm.taskboardId}, // SUCCESS function (response) { console.log(response.data) }, // ERROR function (response) { console.error(response.data) } ); }

Resolve

resolve is a special method that you can use in the state definitions. It will return a promise object and is only available in msApiProvider. msApiProvider.request(Action, Parameters); /** @ngInject */ function config($stateProvider, msApiProvider) { // State $stateProvider.state('app.somepage', { url : '/somepage', views : { // ... }, resolve: { PageData: function (msApi) { return msApi.resolve('app.somepage@get'); } } }); // Api msApiProvider.register('app.somepage', ['http://myapi.com/pageData']); }

Server

As we mentioned before, aap uses Yeoman generator for AngularJS with GulpJS for quickly bootstrapping the application and it comes with a built-in server setup so you don't have to worry about setting up a server for your development environment. The built-in server setup uses browser-sync. There are a lot of benefits of using browser-sync such as testing your app in multiple browser windows and even across multiple devices. Browser-sync will synchronise your actions such as clicking and scrolling across browser windows and devices to make testing easier. To start the server, run the following command in your favorite console application (Terminal, Command Prompt etc.) while in your work folder: gulp serve This will start up the server, compile scss files, inject dependencies, and will do all other related jobs for you. Then it will automatically fire up your default browser and redirect you to the http://localhost:3000 by default. You can check out your console application to get further information about the server. By default, it will run on http://localhost:3000 but it might change depending on your setup.

Reloading the Server

The gulp serve command will watch your files and reload the page for you as you make changes. But for some reason, if you need to manually restart the server, you can do it by pressing Ctrl + C on your keyboard while in the console application and then run the following command once again: gulp serve

What is building?

Once you finish your development, you can build the application. Building process is a combination of couple other processes. Mainly, after building your application, all your html files are going to be injected into a single javascript file and they will become entities in the Angular's $templateCache service. All your style and javascript files will be minified and uglified. And most importantly, you will be able to run your application from a simple host without needing Node.js, Gulp or any other tools we mentioned. Your app will become a simple HTML/Javascript application.

Building

To build your application, simply run the following command in your favorite console application (Terminal, Command Prompt etc.) and wait for it to finish: gulp build And that's it. Now the production version of your app is living under the /dist directory within your work folder. You can directly upload the contents of /dist folder to your server, and it will work. .htaccess

aap uses HTML5 mode in its location service. It means, you will see URLs like www.aap.com/taskboard rather than seeing URLs like this one: www.aap.com/#/taskboard Also, without following .htaccess rules, you can't access your application directly. So typing www.aap.com/taskboard to the url bar or refreshing your application will cause an Internal Server Error. Create an .htaccess that stays with the same directory as your index.html and paste the following code inside it: <IfModule mod_rewrite.c> RewriteEngine On # Send all requests to the index.html unless # it's a directory or a file that actually exists RewriteCond %{REQUEST_FILENAME} !-d RewriteCond %{REQUEST_FILENAME} !-f RewriteRule ^ index.html [L] </IfModule>
